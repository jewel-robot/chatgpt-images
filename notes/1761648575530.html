OUTPUT DROP 是 默认策略（policy） 的一种设置：把 本机发往外界 的所有数据包，在未命中任何允许规则时，一律丢弃。
在 iptables 里，每条“内建链”（INPUT、OUTPUT、FORWARD）都可以设置一个默认策略；这相当于“规则表的兜底结论”。

三条主链与默认策略

  • INPUT：外界 → 本机 的入站流量

  • OUTPUT：本机 → 外界 的出站流量

  • FORWARD：经过本机转发的流量（做路由/NAT 时才会用）

默认策略可设为：

  • ACCEPT：默认放行（白名单越写越多时用得少）

  • DROP：默认丢弃（零信任/最小权限常用）

  • REJECT：显式拒绝并回一个“不可达”（对 TCP/UDP 效果不同，一般面向入站更常见）

  • RETURN：返回到调用者（对用户自定义链有意义，内建链的默认策略不能设 RETURN）

查看/设置：

bash
# 查看默认策略与命中计数
sudo iptables -L -n -v

# 设置默认策略为 DROP（示例）
sudo iptables -P INPUT   DROP
sudo iptables -P OUTPUT  DROP
sudo iptables -P FORWARD DROP

OUTPUT DROP 的含义与影响

  • 含义：除非被你的规则明确允许（例如 --dport 443、-d 192.168.2.3 --dport 1883），否则任何本机发出的包都被丢弃。

  • 影响：

    • 出站“白名单化”，能有效阻止程序私自联网。

    • 需要显式放行一些容易忽略的出站流量，例如：

      • DNS：--dport 53（或走本地 127.0.0.53 则不需）

      • DHCP/DNS 的服务器回复：热点场景要放 --sport 67/53（你刚遇到过）

      • 特定协议的回连/心跳（MQTT、NTP、APT 镜像等）

常见的允许写法（示例）：

bash
# 仅允许这几类出站
iptables -A OUTPUT -p tcp -d 192.168.2.3 --dport 1883 -j ACCEPT
iptables -A OUTPUT -p tcp --dport 28883 -j ACCEPT
iptables -A OUTPUT -p tcp --dport 443   -j ACCEPT
# DNS（若需要外发）
iptables -A OUTPUT -p udp --dport 53 -j ACCEPT
iptables -A OUTPUT -p tcp --dport 53 -j ACCEPT
# 允许已建立/相关（入/出都建议有）
iptables -A OUTPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT  -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

策略组合的几种“流派”

  1. 默认全放行（ACCEPT）+ 黑名单阻断
    易漏，安全性低；只适合临时排障或极简场景。

  2. 默认全丢弃（DROP）+ 白名单放行 ✅
    最安全、最可控；你现在采用的模式。

  3. 入站 DROP，出站 ACCEPT
    服务器常见折中（但程序能随便联网，隐患大）。

  4. 入站 REJECT，出站 DROP
    入站用 REJECT 让合法客户端快速失败（有时便于调试），出站继续严格白名单。

DROP vs REJECT（入站时的选择）

  • DROP：对方看起来像“黑洞”，扫描器会超时重试，信息泄露少。

  • REJECT：立即告诉对方“拒绝”，用户体验好（快速失败），但在某些扫描中更“健谈”。

  • 出站通常不使用 REJECT，直接 DROP 即可。

规则顺序与兜底

iptables 从上到下匹配，第一条命中就停止。因此常见顺序：

  1. lo 回环

  2. ESTABLISHED,RELATED

  3. 业务白名单（按命中频率高的在前）

  4. 最后由默认策略兜底（无需再写显式 DROP）

两个易踩坑

  • 远程操作时把 INPUT 设为 DROP 却忘了放 ssh 会把自己锁外面。可用 iptables-apply 带自动回滚，或先放 --dport 22 再收口。

  • 开了 OUTPUT DROP 却忘了协议的 服务器回复（比如 DHCP/DNS 在热点场景）会被挡，需要放 --sport 67/53。

方案 B：先“插入 SSH 放行”，再收口（原地修改）

在你改默认策略为 DROP 之前，先确保 SSH 放行在最前面：

bash
# 仅当不存在这条时才插入，避免重复
sudo iptables -C INPUT -p tcp --dport 22 -j ACCEPT 2>/dev/null || \
sudo iptables -I INPUT 1 -p tcp --dport 22 -j ACCEPT

# 然后再把默认策略收紧

2) OUTPUT DROP 时，别忘了“服务器回复”的出口

当你的机器充当服务器而且默认 OUTPUT DROP，有几类“必须显式放行的 UDP 回包”：

  • DHCP 服务器（热点）：

    • 客户端请求：客户端:68/udp → 你:67/udp → 你已在 INPUT 放行

    • 你回复：你:67/udp → 客户端:68/udp → 需要显式放行

    bash
    sudo iptables -A OUTPUT -o wlan0 -p udp --sport 67 --dport 68 -j ACCEPT

  • DNS 服务器（热点的 dnsmasq）：

    • 客户端查询：客户端:高端口 → 你:53/udp|tcp → 你已在 INPUT 放行

    • 你答复：你:53/udp → 客户端:高端口（TCP 同理）→ 建议显式放行

    bash
    sudo iptables -A OUTPUT -o wlan0 -p udp --sport 53 -j ACCEPT
# 如需 TCP DNS，也可加一条：
sudo iptables -A OUTPUT -o wlan0 -p tcp --sport 53 -j ACCEPT

  • mosh 服务器（如果你启用）：
    mosh 用 UDP 60000–61000；固定端口举例 60001：

    bash
    # 入站放行客户端到你：--dport 60001
sudo iptables -A INPUT  -p udp -s <client_ip> --dport 60001 -j ACCEPT
# 出站（你回客户端）：--sport 60001
sudo iptables -A OUTPUT -p udp -d <client_ip> --sport 60001 -j ACCEPT

为何要这样做？

  • TCP 的“回复”一般被 -m conntrack --ctstate ESTABLISHED,RELATED 自动放过；

  • UDP 是无连接，尤其 DHCP 的广播/单播混合流程不一定被 conntrack 标记成已建立，所以必须用 --sport <服务器端口> 明确放行第一次应答；DNS 也加上更稳。

一眼自检命令

  • 确保 SSH 永远有口：

bash
sudo iptables -L INPUT -n --line-numbers | sed -n '1,20p'
# 应该能看到很靠前的一条：tcp dpt:22 ACCEPT

  • 观察热点拿 IP 时是否命中 DHCP/DNS 规则：

bash
watch -n1 'sudo iptables -L -n -v | egrep "(wlan0|dpt:(53|67|68)|sport 53|sport 67)"'

以上两套“防锁死”和“UDP 服务器回包口子”按这个做，基本就没有踩坑空间了。
