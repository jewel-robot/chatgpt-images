🐍 Python 示例代码

python
import paho.mqtt.client as mqtt
import serial
import threading
import json

# ===== 配置 =====
MQTT_BROKER = "127.0.0.1"
MQTT_PORT = 1884
MQTT_TOPIC_SUB = "app/#"        # 订阅的主题
MQTT_TOPIC_PUB = "bluetooth/in" # 蓝牙收到数据后发布的主题

SERIAL_PORT = "/dev/rfcomm0"
SERIAL_BAUD = 9600

# ===== 串口对象 =====
ser = serial.Serial(SERIAL_PORT, SERIAL_BAUD, timeout=1)

# ===== MQTT回调 =====
def on_connect(client, userdata, flags, rc):
    print("MQTT 已连接, rc =", rc)
    client.subscribe(MQTT_TOPIC_SUB)

def on_message(client, userdata, msg):
    try:
        payload = msg.payload.decode()
        print(f"MQTT -> 蓝牙: {msg.topic} {payload}")
        ser.write((payload + "\n").encode("utf-8"))
    except Exception as e:
        print("MQTT转发失败:", e)

# ===== 串口监听线程 =====
def serial_listener(client):
    buffer = ""
    while True:
        try:
            data = ser.read().decode("utf-8", errors="ignore")
            if not data:
                continue
            buffer += data
            if "\n" in buffer:
                line, buffer = buffer.split("\n", 1)
                line = line.strip()
                if line:
                    print(f"蓝牙 -> MQTT: {line}")
                    client.publish(MQTT_TOPIC_PUB, line)
        except Exception as e:
            print("串口监听错误:", e)

# ===== 主程序 =====
def main():
    # 启动 MQTT
    client = mqtt.Client()
    client.on_connect = on_connect
    client.on_message = on_message
    client.connect(MQTT_BROKER, MQTT_PORT, 60)

    # 启动串口监听线程
    t = threading.Thread(target=serial_listener, args=(client,))
    t.daemon = True
    t.start()

    # 循环处理 MQTT
    client.loop_forever()

if __name__ == "__main__":
    main()

🐍 修改版桥接程序 (纯 JSON 模式)

python
import paho.mqtt.client as mqtt
import serial
import threading

# ===== 配置 =====
MQTT_BROKER = "127.0.0.1"
MQTT_PORT = 1884
MQTT_TOPIC_SUB = "app/#"        # 接收网页应用发来的
MQTT_TOPIC_PUB = "bluetooth/in" # 蓝牙设备消息发布到这里

SERIAL_PORT = "/dev/rfcomm0"
SERIAL_BAUD = 9600

# ===== 打开串口 =====
ser = serial.Serial(SERIAL_PORT, SERIAL_BAUD, timeout=1)

# ===== MQTT回调 =====
def on_connect(client, userdata, flags, rc):
    print("MQTT 已连接, rc =", rc)
    client.subscribe(MQTT_TOPIC_SUB)

def on_message(client, userdata, msg):
    try:
        payload = msg.payload.decode("utf-8")
        print(f"MQTT -> 蓝牙: {payload}")
        # ✅ 只写入 JSON payload，不包含 MQTT 报文头
        ser.write((payload + "\n").encode("utf-8"))
    except Exception as e:
        print("转发到蓝牙失败:", e)

# ===== 串口监听线程 =====
def serial_listener(client):
    buffer = ""
    while True:
        try:
            data = ser.read().decode("utf-8", errors="ignore")
            if not data:
                continue
            buffer += data
            if "\n" in buffer:
                line, buffer = buffer.split("\n", 1)
                line = line.strip()
                if line:
                    print(f"蓝牙 -> MQTT: {line}")
                    client.publish(MQTT_TOPIC_PUB, line)
        except Exception as e:
            print("串口监听错误:", e)

# ===== 主程序 =====
def main():
    client = mqtt.Client()
    client.on_connect = on_connect
    client.on_message = on_message
    client.connect(MQTT_BROKER, MQTT_PORT, 60)

    t = threading.Thread(target=serial_listener, args=(client,))
    t.daemon = True
    t.start()

    client.loop_forever()

if __name__ == "__main__":
    main()

🔄 双向通信是否会有问题？

1. 可能的循环问题

如果 A 把 MQTT 消息转 JSON 发给 B，B 收到后又把 JSON 转回 MQTT，再发给 A，A 又转回 JSON……就会产生 消息回环。

举个例子：

  • A 上发 {"msg":"hello"} → 桥接 → 蓝牙 → B 收到。

  • B 的桥接又发布到 MQTT → A 的桥接又收回 → 又发蓝牙……死循环。

2. 怎么避免循环？

  • 定义明确的 topic 区分来源

    • 比如 A 桥接只转发 app/# → 蓝牙，收到的蓝牙数据发布到 from_bluetooth/#。

    • B 桥接只转发 control/# → 蓝牙，收到的蓝牙数据发布到 from_bluetooth/#。

    • 这样双方不会“听自己发的东西”。

  • 加上 source 字段

    • 每个 JSON 加 "src":"A" 或 "src":"B"。

    • 桥接收到后先检查，如果是自己发的，就不转发。

👉 推荐用 topic 区分，最直观，网页应用和算法程序也容易区分。

🐍 设备B桥接程序 (1883端口版本)

代码和设备A的类似，只是端口改为 1883，并且加了 topic 区分，避免环路：

python
import paho.mqtt.client as mqtt
import serial
import threading

# ===== 配置 =====
MQTT_BROKER = "127.0.0.1"
MQTT_PORT = 1883
MQTT_TOPIC_SUB = "control/#"       # 设备B订阅的主题
MQTT_TOPIC_PUB = "from_bluetooth"  # 蓝牙收到的数据发布到这里

SERIAL_PORT = "/dev/rfcomm0"
SERIAL_BAUD = 9600

ser = serial.Serial(SERIAL_PORT, SERIAL_BAUD, timeout=1)

# ===== MQTT回调 =====
def on_connect(client, userdata, flags, rc):
    print("MQTT 已连接, rc =", rc)
    client.subscribe(MQTT_TOPIC_SUB)

def on_message(client, userdata, msg):
    try:
        payload = msg.payload.decode("utf-8")
        print(f"MQTT -> 蓝牙: {payload}")
        ser.write((payload + "\n").encode("utf-8"))  # 只发 JSON
    except Exception as e:
        print("转发到蓝牙失败:", e)

# ===== 串口监听线程 =====
def serial_listener(client):
    buffer = ""
    while True:
        try:
            data = ser.read().decode("utf-8", errors="ignore")
            if not data:
                continue
            buffer += data
            if "\n" in buffer:
                line, buffer = buffer.split("\n", 1)
                line = line.strip()
                if line:
                    print(f"蓝牙 -> MQTT: {line}")
                    client.publish(MQTT_TOPIC_PUB, line)
        except Exception as e:
            print("串口监听错误:", e)

# ===== 主程序 =====
def main():
    client = mqtt.Client()
    client.on_connect = on_connect
    client.on_message = on_message
    client.connect(MQTT_BROKER, MQTT_PORT, 60)

    t = threading.Thread(target=serial_listener, args=(client,))
    t.daemon = True
    t.start()

    client.loop_forever()

if __name__ == "__main__":
    main()

🔍 通信逻辑总结

  • 设备A：

    • app/# → JSON → 蓝牙

    • 蓝牙收到 → 发布到 from_bluetooth/#

  • 设备B：

    • control/# → JSON → 蓝牙

    • 蓝牙收到 → 发布到 from_bluetooth/#

这样：

  • 网页应用只发 app/test，设备B算法只看 control/test；

  • 彼此转发不会混淆，也不会无限循环。

你说的情况是这样的：

  • A 和 B 之间通过 蓝牙 RFCOMM 建立了一个虚拟串口连接。

  • 在 A 上这个串口表现为 /dev/rfcomm0，在 B 上也有一个 /dev/rfcomm0（或者别的号）。

📌 串口通信的本质

  • 串口（包括蓝牙 RFCOMM 映射的虚拟串口）是 点对点的全双工通道。

  • 它的行为就像两台电脑之间拉了一根串口线：

    • A 发送的数据会出现在 B 的串口接收端；

    • B 发送的数据会出现在 A 的串口接收端。

  • 不会出现 A 发出去的数据，又从 A 自己的串口读出来。

换句话说，串口连接不是广播，而是点对点：你发出去的字节只会到对方，不会“回声”到自己。

🔄 那双向通信会不会有问题？

不会。RFCOMM 就像一根双向的串口线：

  • 如果 A 写 /dev/rfcomm0，数据到 B。

  • 如果 B 写 /dev/rfcomm0，数据到 A。

  • A 和 B 各自收到的，都是对方发来的，不会收到自己发的。

⚠️ 唯一的例外：

  • 如果你代码里写错了，比如 同时一个程序在写又在读同一个 buffer，可能自己把自己发的数据读出来（这种是应用层 bug，不是串口的特性）。

  • 或者蓝牙驱动配置了“本地回显”（一般不会）。

✅ 总结

  • 串口（RFCOMM）是点对点的，不会自回显。

  • A 发的，只能被 B 收到；B 发的，只能被 A 收到。

  • 所以你完全可以在 A 和 B 上各跑一个桥接程序，双向通信不会互相干扰，也不会形成死循环。

ChatGPT:

这个现象我大概能猜出来 👀：

你在设备 A 上执行了

bash
cat /dev/rfcomm0

结果终端马上就退出了，没有像预期那样阻塞等待。

📌 原因可能有几个

1. 蓝牙连接没建立好

  • /dev/rfcomm0 这个设备文件是存在的，但如果 A 没有成功连接到 B，那么一旦你去读，它会立刻返回 EOF（文件结束）。

  • 对 cat 来说，读到 EOF 就意味着“文件结束”，所以直接退出。

2. 设备B没有在监听

如果 A 是 client，但 B 上没有运行 rfcomm listen 或者你的桥接程序没有打开串口，那么 A 打开 /dev/rfcomm0 会失败或立刻关闭。

3. 连接被另一进程占用

如果 /dev/rfcomm0 已经被某个程序（比如你的 Python bridge）占用了，再 cat 可能会行为异常。

✅ 建议验证步骤

1. 确认连接

在设备 A 上执行：

bash
rfcomm

看看 /dev/rfcomm0 的状态。如果显示 disconnected，那肯定没连上。

2. 确保 B 在监听

在设备 B 上执行：

bash
sudo rfcomm listen /dev/rfcomm0 1

保持它阻塞监听。

3. 在 A 上手动连接

在设备 A 上执行：

bash
sudo rfcomm connect /dev/rfcomm0 <B的蓝牙MAC地址> 1

等提示 Connected 之后，再试：

bash
cat /dev/rfcomm0

这时候就会阻塞，等 B 发消息。

4. 如果还是退出

说明 B 没有发数据并且连接立即关闭。要么是 B 端进程退出了，要么是蓝牙链路掉了。

🔑 总结

  • cat /dev/rfcomm0 会退出，是因为串口设备返回了 EOF。

  • 最常见的原因：蓝牙连接没建立 / B 没监听。

  • 正常情况下，如果连接成功，cat 会阻塞等待，不会直接结束。
