1. ROS 可配置参数

这些参数是在构造函数里通过 pnh_.param(...) 读进来的。也就是你最可能在 launch 或 yaml 里改的那一批。

1.2 采样与滤波参数

num_frames_

对应参数名：

C++
num_frames

默认值：

C++
30

含义：

  • 需要累计多少帧点云和多少帧激光，才开始优化。

作用：

  • cloud_count_ >= num_frames_

  • scan_count_ >= num_frames_
    两者都满足时才开始标定。

经验上：

  • 大一点更稳

  • 太大则采集时间更长，也可能把场景变化引入进来

voxel_leaf_

对应参数名：

C++
voxel_leaf

默认值：

C++
0.02

单位：

  • 米

含义：

  • 点云体素降采样的体素边长。

作用：

  • VoxelGrid 下采样时 x/y/z 三个方向都用这个值。

直观理解：

  • 越大，点更稀疏，计算更快，但平面精度可能下降

  • 越小，细节更多，但噪声和计算量也更大

z_min_, z_max_

对应参数名：

C++
z_min
z_max

默认值：

C++
0.1
5.0

单位：

  • 米

注意：
虽然变量名叫 z_min/z_max，但这里实际不是按 z 轴裁剪，而是按“点到传感器原点的欧氏距离”做筛选。代码里是算 dist = sqrt(dx²+dy²+dz²)。

含义：

  • 只保留离传感器距离在 [z_min_, z_max_] 范围内的点。

作用：

  • 去掉太近的噪声点和太远的不可靠点。

这块变量名略有误导，严格说更像：

  • range_min

  • range_max

3. 数据结构里的字段

3.1 PlaneParam

定义：

C++
struct PlaneParam {
    Eigen::Vector3d normal;
    double d;
    int inlier_count;
};

normal

含义：

  • 平面法向量 (nx, ny, nz)，通常已归一化。

平面方程：

nxx+nyy+nzz+d=0n_x x + n_y y + n_z z + d = 0nx​x+ny​y+nz​z+d=0

d

含义：

  • 平面方程中的常数项。

注意：

  • 这个量会随着法向量翻转一起翻转：

    • normal -> -normal

    • d -> -d
      代码里确实这样做了。

inlier_count

含义：

  • 拟合这个平面的内点数量。

作用：

  • 后面平均法向量时拿它做权重。

3.2 LineParam

定义：

C++
struct LineParam {
    Eigen::Vector2d direction;
    double normal_azimuth;
    int inlier_count;
};

direction

含义：

  • 2D 直线方向向量，单位向量。

normal_azimuth

含义：

  • 直线法向量在 x-y 平面内的方位角：

atan2(ny,nx)atan2(n_y, n_x)atan2(ny​,nx​)

作用：

  • 这是 wall azimuth cost 的目标量。

inlier_count

含义：

  • 拟合出这条线时的内点数量。

作用：

  • 作为 scan 质量门槛。

4. 优化相关参数

这一部分最关键。

4.1 rpy

在 runCalibration() 里：

C++
double rpy[3] = { initial_rpy_.x(), initial_rpy_.y(), initial_rpy_.z() };

含义：

  • 优化变量，长度 3：

    • rpy[0] = roll

    • rpy[1] = pitch

    • rpy[2] = yaw

它表示的是：

  • 最终待估计的 camera_frame -> base_frame 旋转

不是增量，而是最终旋转本身。代码注释也直接这么写了。

4.2 ground_normal_cam

C++
Eigen::Vector3d ground_normal_cam = averageNormals(ground_planes_, "ground");

含义：

  • 多帧平均后的地面法向量，位于 camera frame。

作用：

  • GroundNormalCost 的输入。

4.3 wall_normal_cam

含义：

  • 多帧平均后的墙法向量，位于 camera frame。

作用：

  • WallAzimuthCost 和 WallVerticalityCost 的输入。

4.4 lidar_wall_azimuth

含义：

  • 多帧平均后的激光墙法向方位角，位于 base_link 的 x-y 平面。

作用：

  • WallAzimuthCost 的目标值。

4.5 roll_opt, pitch_opt, yaw_opt

含义：

  • 优化后的最终欧拉角结果。

它们会再转回旋转矩阵 R_opt，以及四元数 q_opt。

4.6 R_opt

含义：

  • 由最终 roll_opt/pitch_opt/yaw_opt 组成的旋转矩阵。

作用：

  • 用来验证：

    • 地面是否被转正到 z 轴

    • 墙面是否水平法向正确

    • 墙方位角是否与 lidar 一致

5. 三个 cost 里的参数

5.1 GroundNormalCost

定义里只有一个成员：

n_ground_cam

含义：

  • 地面法向量，位于 camera frame。

operator() 里的局部量：

cr, sr

  • cos(roll), sin(roll)

cp, sp

  • cos(pitch), sin(pitch)

cy, sy

  • cos(yaw), sin(yaw)

R00 ~ R22

  • 手动展开的旋转矩阵元素，对应

R=Rz(yaw)Ry(pitch)Rx(roll)R = R_z(yaw)R_y(pitch)R_x(roll)R=Rz​(yaw)Ry​(pitch)Rx​(roll)

nx, ny, nz

  • 输入法向量的三个分量

rx, ry, rz

  • 旋转后的法向量分量，即

R⋅nground_camR \cdot n_{ground\_cam}R⋅nground_cam​

residual[0], residual[1], residual[2]

  • 残差，目标是让旋转后的法向量变成 [0,0,1]。

5.2 WallAzimuthCost

n_wall_cam

  • 相机系下墙法向量。

lidar_azimuth

  • 激光测得的墙法向角，单位 rad。

局部量里：

cam_az

  • 旋转后墙法向量在 base_link x-y 平面的方位角：

atan2(ry,rx)atan2(ry, rx)atan2(ry,rx)

diff

  • 相机推导出的墙方位角，与激光方位角的差

residual[0] = sin(diff)

  • 用 sin 而不是直接 diff，是为了处理角度绕回问题。

5.3 WallVerticalityCost

n_wall_cam

  • 相机系下墙法向量。

局部量里：

R20, R21, R22

  • 旋转矩阵的第三行

rz

  • 旋转后墙法向量的 z 分量

residual[0] = rz

  • 希望墙法向量 z 分量接近 0，也就是墙是竖直平面。

6. 容易混淆的几个参数

这里我专门指出几个最容易误解的。

z_min_ / z_max_

名字像按 z 轴裁剪，实际上是按点到传感器的距离筛选。

initial_quaternion_ / initial_rpy_

它们不是优化结果，而是：

  • 从现有 TF 树中读出的初始外参旋转

  • 给优化提供初值和法向量回转参考。

weight_ground_ / weight_wall_az_ / weight_wall_vert_

它们不是直接乘残差，而是通过 ScaledLoss 加进去，控制三类约束的相对重要性。

ground_planes_ / wall_planes_

虽然平面是一开始在 base_link 下拟合出来的，但真正存进去用于优化的法向量，已经被转回 camera frame 了。

好，我们就从 cloudCallback() 开始，按代码顺序一段一段、尽量逐句讲。这个函数的职责可以先概括成一句：

收到一帧深度点云后，把它变到 base_link，清洗和下采样，然后拟合地面和平面墙，把这两个平面的法向量转回 camera frame 保存起来，供后面的 Ceres 优化使用。

我会按原函数的结构往下拆。

1. 函数签名

C++
void cloudCallback(const sensor_msgs::PointCloud2::ConstPtr& msg)

含义：

  • 这是 ROS 的点云回调函数。

  • 每收到一条 sensor_msgs::PointCloud2 消息，就会执行一次。

  • msg 是一个常量智能指针，指向收到的点云消息。

这里的 msg 里最重要的几个东西是：

  • msg->header.frame_id：这帧点云当前所在的坐标系

  • msg->header.stamp：时间戳

  • 点云数据本体

这也是后面为什么会先检查 frame_id，再做 TF 变换。

2. 加锁，防止并发访问共享数据

C++
std::lock_guard<std::mutex> lk(mtx_);

含义：

  • 进入回调后，先把互斥锁 mtx_ 锁住。

  • 这个函数会访问很多共享成员变量，比如：

    • cloud_count_

    • ground_planes_

    • wall_planes_

    • debug_log_

    • calibration_done_

因为 ROS 可能让不同回调并发执行，所以这里加锁是为了防止：

  • 点云回调和激光回调同时修改共享变量

  • 或者一个回调还没写完，另一个又读到了中间状态

std::lock_guard 是 RAII 风格：函数退出时会自动解锁，不需要你手动 unlock()。

3. 如果已经标定完成，就直接退出

C++
if (calibration_done_) return;

含义：

  • 一旦整个标定流程已经结束，后续再来的点云就不再处理。

原因：

  • 这份代码不是持续在线优化，而是“收集 N 帧 → 标定一次 → 结束采集”。

所以 calibration_done_ == true 之后，点云只可能用于后面的可视化模式，不再进入这个采集回调。

4. 如果已经收够点云帧数，也直接退出

C++
if (cloud_count_ >= num_frames_) return;

含义：

  • num_frames_ 是你配置的目标帧数，比如 30。

  • 如果已经收集够这么多“有效点云帧”，就不再继续累计。

注意这里是“有效帧”，不是“收到的帧”。
因为很多帧可能会由于：

  • TF 查不到

  • 点太少

  • 拟合失败

  • frame_id 不匹配
    而被提前丢弃。

只有最后真正 push_back 成功并 cloud_count_++ 的，才算一帧有效样本。

5. 检查点云 frame_id 是否符合预期

C++
// P2: frame_id check
if (!expected_cloud_frame_.empty() &&
    msg->header.frame_id != expected_cloud_frame_) {
    ROS_WARN_THROTTLE(5.0,
        "Cloud frame_id mismatch: expected '%s', got '%s'. Skipping.",
        expected_cloud_frame_.c_str(), msg->header.frame_id.c_str());
    return;
}

这一段的逻辑是：

5.1 !expected_cloud_frame_.empty()

表示：

  • 如果你没有配置 expected_cloud_frame，那就不做检查

  • 只有你显式设置了预期 frame，才启用这个保护

5.2 msg->header.frame_id != expected_cloud_frame_

表示：

  • 当前收到的点云 frame 跟你预期的不一致

比如你期望是：

  • left_rgbd_optical_frame

但实际收到的是：

  • left_rgbd_link

那就说明你的话题或发布链路可能不对。

5.3 ROS_WARN_THROTTLE(5.0, ...)

含义：

  • 打 warning，但最多每 5 秒打印一次

  • 避免高频话题把终端刷爆

5.4 return

发现 frame 不对，这一帧直接跳过。

这一步的作用是做“输入合法性保护”。
因为如果点云 frame 搞错了，后面的 TF、平面法向量方向、标定结果都会跟着错。

6. 第一次进入时，先读取初始 TF

C++
// Read initial TF once
if (!got_initial_tf_) {
    if (!readInitialTF()) return;
}

含义：

  • got_initial_tf_ 表示是否已经成功读取过一次初始外参

  • 如果还没读，就调用 readInitialTF()

readInitialTF() 做的事是：

  • 从 TF 树读取 base_frame_ -> camera_frame_

  • 保存平移到 initial_translation_

  • 保存旋转到 initial_quaternion_

  • 再转换成 initial_rpy_

如果读取失败，就 return，这一帧也不处理。

这一步非常关键，因为后面：

  • 点云处理是在 base_link 下做的

  • 但最后存储给优化器的是 camera frame 下的法向量

  • 所以要用这个初始旋转把法向量从 base_link 转回 camera frame

也就是说，初始 TF 是整个 refinement 的参考基准。

7. 记住点云原始 frame，供后续可视化使用

C++
// Remember cloud's native frame for visualization
if (cloud_native_frame_.empty()) {
    cloud_native_frame_ = msg->header.frame_id;
}

含义：

  • cloud_native_frame_ 只在第一次收到有效点云时记录下来

  • 记录的是点云消息原本的 frame_id

用途：

  • 标定完成后，可视化代码需要知道“原始点云来自哪个 frame”

  • 后面 vizCloudCallback() 会用它构造“原始 TF”和“校正后 TF”的显示链路

所以这里不是标定本身必需，而是为后面的 RViz 调试准备。

8. 把 ROS 点云消息转成 PCL 点云

C++

pcl::PointCloud<pcl::PointXYZ>::Ptr raw(new pcl::PointCloud<pcl::PointXYZ>);
pcl::fromROSMsg(*msg, *raw);

含义：

  • 先创建一个 PCL 点云对象 raw

  • 类型是 pcl::PointXYZ，即每个点只有 (x,y,z) 三个浮点坐标

  • 再把 ROS 的 PointCloud2 转成 PCL 格式

为什么要转？

因为后面你要用的是 PCL 的：

  • 体素滤波

  • 平面分割

  • 点提取
    这些都基于 pcl::PointCloud<>。

9. 如果原始点云为空，直接跳过

C++
if (raw->empty()) {
    ROS_WARN_THROTTLE(2.0, "Empty cloud received");
    return;
}

含义：

  • 如果这一帧消息里压根没有点，就没必要继续了

这里用 2.0 秒节流打印 warning，原因同前。

10. 去掉 NaN / Inf 点

C++
// Remove NaN
pcl::PointCloud<pcl::PointXYZ>::Ptr cloud(new pcl::PointCloud<pcl::PointXYZ>);
cloud->reserve(raw->size());
for (const auto& p : raw->points) {
    if (std::isfinite(p.x) && std::isfinite(p.y) && std::isfinite(p.z)) {
        cloud->push_back(p);
    }
}

这一段非常常见，也很重要。

10.1 创建新点云 cloud

C++
pcl::PointCloud<pcl::PointXYZ>::Ptr cloud(new pcl::PointCloud<pcl::PointXYZ>);

作用：

  • 新建一个“清洗后的点云”

10.2 cloud->reserve(raw->size());

作用：

  • 预先分配足够内存

  • 避免 push_back 时反复扩容，提高效率

注意：

  • reserve() 只分配容量，不改变实际元素个数

10.3 遍历所有原始点

C++
for (const auto& p : raw->points)

逐点检查。

10.4 std::isfinite(...)

只有当 x/y/z 都是有限数时，才认为这个点有效。

因为深度相机点云经常会出现：

  • NaN

  • Inf

  • 无效深度投影出来的异常值

这些点如果不去掉，后面：

  • TF 变换

  • 距离计算

  • RANSAC 拟合
    都可能出问题。

10.5 cloud->push_back(p);

把有效点放入清洗后的点云。

11. 如果有效点太少，也跳过

C++
if (cloud->size() < 100) {
    ROS_WARN_THROTTLE(2.0, "Too few valid points (%lu)", cloud->size());
    return;
}

含义：

  • 清洗完之后，如果点少于 100 个，说明这一帧质量太差

为什么要这样做？

因为后面要拟合：

  • 地面平面

  • 墙面平面

如果点很少，平面拟合基本不可靠，继续做只会把坏样本引进来。

12. 查 TF：把点云从原始 frame 变到 base_link

C++
// ----------------------------------------------------------
// Transform cloud from its native frame to base_link.
// The data is in msg->header.frame_id (e.g. camera optical
// frame), NOT necessarily base_link.
// ----------------------------------------------------------
Eigen::Affine3d T_base_cloud;
try {
    geometry_msgs::TransformStamped tf_cloud = tf_buffer_.lookupTransform(
        base_frame_, msg->header.frame_id, ros::Time(0), ros::Duration(1.0));
    T_base_cloud = tf2::transformToEigen(tf_cloud);
} catch (tf2::TransformException& ex) {
    ROS_WARN_THROTTLE(5.0, "Cannot transform cloud from '%s' to '%s': %s",
                      msg->header.frame_id.c_str(), base_frame_.c_str(), ex.what());
    return;
}

这一段是整个函数最核心的坐标系转换。

12.1 Eigen::Affine3d T_base_cloud;

定义一个仿射变换：

Tbase←cloudT_{\text{base}\leftarrow\text{cloud}}Tbase←cloud​

含义是：

  • 把 cloud frame 下的点，变换到 base frame

12.2 lookupTransform(base_frame_, msg->header.frame_id, ...)

这句非常关键。

tf2 的语义是：

C++
lookupTransform(target, source)

返回的是：

  • 从 source 到 target 的变换

所以这里拿到的是：

Tbase←cloudT_{\text{base}\leftarrow\text{cloud}}Tbase←cloud​

正好符合后面要做的事。

12.3 ros::Time(0)

表示：

  • 取最新可用变换

12.4 ros::Duration(1.0)

表示：

  • 最多等 1 秒

12.5 tf2::transformToEigen(tf_cloud)

把 ROS 的 TF 消息转成 Eigen 里的仿射变换对象，便于后面直接乘向量。

12.6 catch (...)

如果 TF 查不到，比如：

  • TF 树断了

  • frame 名写错

  • 发布延迟太大

那这一帧直接丢弃。

13. 对点云中的每个点应用这个 TF

C++
for (auto& p : cloud->points) {
    Eigen::Vector3d pt(p.x, p.y, p.z);
    pt = T_base_cloud * pt;
    p.x = pt.x(); p.y = pt.y(); p.z = pt.z();
}

含义：

  • 遍历清洗后的每个点

  • 把点从原始点云 frame 变到 base_link

具体逻辑：

13.1 Eigen::Vector3d pt(p.x, p.y, p.z);

先把 PCL 点转成 Eigen 三维向量。

13.2 pt = T_base_cloud * pt;

做刚体变换：

pbase=Tbase←cloud⋅pcloudp_{base} = T_{\text{base}\leftarrow\text{cloud}} \cdot p_{cloud}pbase​=Tbase←cloud​⋅pcloud​

这里包括：

  • 旋转

  • 平移

13.3 再写回 p

这样处理完后，cloud->points 里的所有点就都已经处于 base_link 下了。

这一步后面的意义很大：

  • 地面“朝上”是相对于 base_link 的 z 轴说的

  • 墙“竖直”也是相对于 base_link 的 z 轴说的

所以要先把点云变到 base_link，后面的约束才有物理意义。

14. 基于传感器原点做距离筛选

C++
// Passthrough: distance filter relative to sensor origin.
// The sensor origin in base_link is the translation part of the
// cloud-frame -> base_link transform.
pcl::PointCloud<pcl::PointXYZ>::Ptr filtered(new pcl::PointCloud<pcl::PointXYZ>);
{
    Eigen::Vector3d sensor_pos = T_base_cloud.translation();
    double cx = sensor_pos.x();
    double cy = sensor_pos.y();
    double cz = sensor_pos.z();
    filtered->reserve(cloud->size());
    for (const auto& p : cloud->points) {
        double dx = p.x - cx, dy = p.y - cy, dz = p.z - cz;
        double dist = std::sqrt(dx*dx + dy*dy + dz*dz);
        if (dist >= z_min_ && dist <= z_max_) {
            filtered->push_back(p);
        }
    }
}

这一段很值得仔细看，因为变量名有点迷惑。

14.1 filtered

创建一个新点云，用来存“距离过滤后”的点。

14.2 sensor_pos = T_base_cloud.translation();

这句的意思是：

  • T_base_cloud 是 cloud frame 到 base 的变换

  • 它的 translation 部分，就是“cloud frame 原点在 base_link 中的位置”

而 cloud frame 原点通常就对应相机传感器原点。

所以：

sensor\_pos = \text{相机原点在 base_link 下的位置}

14.3 cx, cy, cz

只是把这个传感器位置拆成三个标量，方便后面算距离。

14.4 遍历每个点并计算相对传感器的欧氏距离

C++
double dx = p.x - cx, dy = p.y - cy, dz = p.z - cz;
double dist = std::sqrt(dx*dx + dy*dy + dz*dz);

注意这里不是按 z 坐标过滤。
而是算：

点到相机原点的三维欧氏距离

所以虽然参数名叫 z_min_ / z_max_，但实际语义更接近：

  • range_min

  • range_max

14.5 在 [z_min_, z_max_] 内才保留

C++
if (dist >= z_min_ && dist <= z_max_)

这样会去掉：

  • 太近的点（可能是传感器近场噪声）

  • 太远的点（深度误差变大、点云太稀）

14.6 filtered->push_back(p);

把满足距离条件的点留下来。

这一步能明显改善后面的平面拟合稳定性。

15. 对过滤后的点云做体素下采样

C++
// Voxel downsample
pcl::PointCloud<pcl::PointXYZ>::Ptr down(new pcl::PointCloud<pcl::PointXYZ>);
{
    pcl::VoxelGrid<pcl::PointXYZ> vg;
    vg.setInputCloud(filtered);
    vg.setLeafSize(voxel_leaf_, voxel_leaf_, voxel_leaf_);
    vg.filter(*down);
}

含义：

  • 用 PCL 的 VoxelGrid 对点云降采样

原理：

  • 把空间划分成很多边长为 voxel_leaf_ 的小立方体

  • 每个立方体里只保留一个代表点

  • 从而减少点数量，保留大体几何形状

15.1 为什么要降采样？

因为深度点云往往点很多，直接拿来做 RANSAC：

  • 计算更慢

  • 局部噪声更明显

  • 平面内重复密集点太多

做完体素下采样后：

  • 速度更快

  • 大平面结构还在

  • 对标定这种“只关心平面方向”的任务很合适

15.2 setLeafSize(voxel_leaf_, voxel_leaf_, voxel_leaf_)

表示 x、y、z 三个方向体素边长相同。

16. 下采样后再做一次点数检查

C++
if (down->size() < 100) {
    ROS_WARN_THROTTLE(2.0, "Too few points after downsampling (%lu)", down->size());
    return;
}

含义：

  • 即使清洗前有效点很多，经过距离筛选和体素滤波后，仍可能变得太少

  • 若少于 100 个点，这帧就不值得继续做平面拟合

这是第二层质量门槛。

17. 构造“初始旋转”和它的逆

C++
// R_init = rotation part of the camera_frame -> base_link TF.
// After the cloud-frame transform above, the point cloud (and
// therefore the fitted normals) are in base_link.
// We need R_init^(-1) to transform normals back to camera frame (P0).
Eigen::Matrix3d R_init = initial_quaternion_.toRotationMatrix();
Eigen::Matrix3d R_init_inv = R_init.transpose();

这一段是整个函数最容易混淆、但又最关键的地方。

17.1 R_init

它来自 initial_quaternion_，而 initial_quaternion_ 是由：

C++
lookupTransform(base_frame_, camera_frame_, ...)

读出来的。

所以 R_init 表示的是：

Rbase←camR_{\text{base}\leftarrow\text{cam}}Rbase←cam​

也就是：

  • 把 camera frame 下的向量旋转到 base_link

17.2 为什么现在要它的逆？

因为此时点云已经被变换到了 base_link，所以后面拟合出来的平面法向量也是在 base_link 下。

但优化器里的 cost 是按这种形式写的：

Rbase←cam⋅ncamR_{\text{base}\leftarrow\text{cam}} \cdot n_{cam}Rbase←cam​⋅ncam​

也就是说，它希望输入的是 camera frame 下的法向量。

因此这里要把平面法向量从 base_link 再转回 camera frame：

ncam=Rbase←cam−1 nbasen_{cam} = R_{\text{base}\leftarrow\text{cam}}^{-1} \, n_{base}ncam​=Rbase←cam−1​nbase​

17.3 为什么逆矩阵写成转置？

因为旋转矩阵是正交矩阵：

R−1=RTR^{-1} = R^TR−1=RT

所以：

C++
R_init_inv = R_init.transpose();

这样更高效。

18. 设置地面/墙面拟合的角度容差和参考轴

C++
// ----- Fit ground: SACMODEL_PERPENDICULAR_PLANE with axis Z -----
// This finds the largest plane whose normal is within eps_angle of [0,0,1].
const double ground_eps = 30.0 * M_PI / 180.0; // 30° tolerance
const double wall_eps   = 30.0 * M_PI / 180.0; // 30° tolerance
const Eigen::Vector3f z_axis(0.0f, 0.0f, 1.0f);

含义：

  • ground_eps：地面拟合时，允许平面法向量偏离 z 轴最多 30°

  • wall_eps：墙面拟合时，允许相对于 z 轴的约束偏差最多 30°

  • z_axis：参考轴 [0,0,1]

注意，这里的参考轴定义都是在 base_link 下的，因为点云已经在 base_link 里了。

19. 拟合地面平面

C++
PlaneParam ground;
pcl::PointCloud<pcl::PointXYZ>::Ptr after_ground(new pcl::PointCloud<pcl::PointXYZ>);
pcl::PointCloud<pcl::PointXYZ>::Ptr ground_inliers;
if (!fitConstrainedPlane(down, pcl::SACMODEL_PERPENDICULAR_PLANE,
                         z_axis, ground_eps,
                         ground, after_ground, &ground_inliers)) {
    ROS_WARN_THROTTLE(2.0, "Failed to fit ground plane (PERPENDICULAR_PLANE to Z)");
    return;
}

这一段是“找地面”。

19.1 PlaneParam ground;

定义一个结构体，用来接收拟合出来的地面参数：

  • ground.normal

  • ground.d

  • ground.inlier_count

19.2 after_ground

这是一个新点云，用来存：

把地面内点剔除后，剩下的点

目的是后面拿它去找墙。

19.3 ground_inliers

这是指向“地面内点云”的指针，供调试发布给 RViz 用。

19.4 fitConstrainedPlane(...)

调用约束平面拟合函数。

参数逐个解释：

  • down：输入点云，已经清洗、变换、滤波、下采样

  • pcl::SACMODEL_PERPENDICULAR_PLANE：模型类型

  • z_axis：参考轴

  • ground_eps：角度容差

  • ground：输出平面参数

  • after_ground：输出剩余点云

  • &ground_inliers：输出内点云

为什么是 PERPENDICULAR_PLANE？

在 PCL 的这个模型语义里，这里想找的是：

法向量接近 z 轴的平面

这正对应“地面”：

  • 地面平面的法向量应当朝上/朝下

  • 即与 z 轴平行

代码注释里也明确说了：它找的是“法向量在 eps_angle 内接近 [0,0,1] 的最大平面”。

19.5 如果拟合失败，就直接跳过这帧

比如：

  • 地面点不够

  • 场景里根本没看到地面

  • 初始 TF 太差，导致在 base_link 中地面不像地面

那这一帧样本就没法用了。

20. 统一地面法向量方向：朝上

C++
// Orient ground normal upward in base_link (z > 0)
if (ground.normal.z() < 0) {
    ground.normal = -ground.normal;
    ground.d      = -ground.d;
}

这一步是处理平面法向量的“符号二义性”。

因为平面法向量本来就有两种等价表示：

n,d和−n,−dn, d \quad \text{和} \quad -n, -dn,d和−n,−d

表示的是同一个平面。

但如果你不统一方向，不同帧可能有的朝上、有的朝下，后面平均法向量就会互相抵消。

所以这里强制规定：

地面法向量在 base_link 中必须满足 z > 0

也就是“朝上”。

如果当前 z < 0，就把整个平面参数翻转。

21. 把地面法向量从 base_link 转回 camera frame

C++
// P0: Transform ground normal to camera frame
ground.normal = (R_init_inv * ground.normal).normalized();

这一句的数学含义是：

ngroundcam=Rbase←cam−1 ngroundbasen_{ground}^{cam} = R_{\text{base}\leftarrow\text{cam}}^{-1} \, n_{ground}^{base}ngroundcam​=Rbase←cam−1​ngroundbase​

为什么要这么做？

因为后面 Ceres cost 是按“输入 camera frame 下的法向量”来建模的。
所以虽然地面是在 base_link 下拟合出来的，但保存前要转回相机系。

.normalized() 表示再归一化一次，保证长度为 1，避免数值误差影响。

22. 拟合墙面平面

C++
// ----- Fit wall: SACMODEL_PARALLEL_PLANE with axis Z -----
// This finds the largest plane whose normal is perpendicular to [0,0,1]
// (i.e. the plane itself is parallel to Z axis = vertical wall).
PlaneParam wall;
pcl::PointCloud<pcl::PointXYZ>::Ptr after_wall(new pcl::PointCloud<pcl::PointXYZ>);
pcl::PointCloud<pcl::PointXYZ>::Ptr wall_inliers;
if (!fitConstrainedPlane(after_ground, pcl::SACMODEL_PARALLEL_PLANE,
                         z_axis, wall_eps,
                         wall, after_wall, &wall_inliers)) {
    ROS_WARN_THROTTLE(2.0, "Failed to fit wall plane (PARALLEL_PLANE to Z)");
    return;
}

这一段是在“去掉地面后，从剩余点云中找墙”。

22.1 为什么输入是 after_ground 而不是 down？

因为你已经把地面找出来了。
如果还在全点云里找，第二次 RANSAC 很可能又会偏向大地面。

所以这里先剔除地面，再找“次大平面”，更容易找到墙。

22.2 SACMODEL_PARALLEL_PLANE

这里的几何目标是：

找一个与 z 轴平行的平面，也就是“竖直墙面”

竖直墙面的特点是：

  • 平面本身沿 z 方向竖起来

  • 因此它的法向量应当垂直 z 轴

  • 也就是法向量主要落在 x-y 平面里

代码注释也正是这么写的。

22.3 after_wall

这里虽然输出了“去掉墙面后的剩余点云”，但在当前函数里没再继续用。
更像是为了让 fitConstrainedPlane() 接口统一。

22.4 wall_inliers

墙面内点云，用于 RViz 调试显示。

23. 把墙面法向量也转回 camera frame

C++
// P0: Transform wall normal to camera frame
wall.normal = (R_init_inv * wall.normal).normalized();

和地面同理：

  • 墙面最初是在 base_link 下拟合出来的

  • 但优化器希望输入 camera frame 下的法向量

  • 所以这里也要逆旋转回去

公式同样是：

nwallcam=Rbase←cam−1 nwallbasen_{wall}^{cam} = R_{\text{base}\leftarrow\text{cam}}^{-1} \, n_{wall}^{base}nwallcam​=Rbase←cam−1​nwallbase​

24. 检查墙面和地面是否近似垂直

C++
// Verify perpendicularity in camera frame (should hold since rotation preserves angles)
{
    double dot = std::abs(wall.normal.dot(ground.normal));
    if (dot > perp_dot_thresh_) {
        ROS_WARN_THROTTLE(2.0, "Wall not perpendicular to ground (dot=%.3f). Skipping.", dot);
        return;
    }
}

这一段是一个数据质量检查。

24.1 wall.normal.dot(ground.normal)

点积表示两法向量夹角关系：

  • 如果垂直，点积应接近 0

  • 如果平行，绝对值接近 1

这里由于墙和地面本应互相垂直，所以它们的法向量也应垂直。

24.2 std::abs(...)

因为法向量可能差一个负号，所以看绝对值更合理。

24.3 if (dot > perp_dot_thresh_)

如果绝对点积太大，说明：

  • 找到的墙不太像墙

  • 或地面拟合不准

  • 或场景本身不是“墙-地面正交”

  • 或 TF 初始值有问题，导致在 base_link 中分割语义不成立

那这一帧直接丢弃。

24.4 为什么在 camera frame 检查也可以？

因为旋转不改变夹角。
法向量从 base_link 转回 camera frame 后，它们的夹角仍然相同。

所以在 camera frame 下检查正交性也成立。

25. 统一墙法向量的符号方向

C++
// P1: Unify wall normal sign across frames.
// Use the first frame's wall normal as reference direction.
if (!have_first_wall_normal_) {
    first_wall_normal_cam_ = wall.normal;
    have_first_wall_normal_ = true;
} else {
    if (wall.normal.dot(first_wall_normal_cam_) < 0) {
        wall.normal = -wall.normal;
        wall.d      = -wall.d;
    }
}

这一步非常重要。

25.1 为什么只对墙做“跨帧方向统一”？

地面前面已经强制成 z > 0 了，所以方向天然统一。
但墙法向量在水平面里，没有“朝上/朝下”这种天然基准，所以必须额外处理。

25.2 第一帧墙法向量作为参考

C++
if (!have_first_wall_normal_)

如果这是第一次成功检测到墙，就把当前方向记下来：

  • first_wall_normal_cam_ = wall.normal

  • 并设置 have_first_wall_normal_ = true

25.3 后续帧和第一帧比较

C++
if (wall.normal.dot(first_wall_normal_cam_) < 0)

如果点积小于 0，说明当前法向量和第一帧方向相反，大约差 180°。

那就翻转它：

  • wall.normal = -wall.normal

  • wall.d = -wall.d

这样所有帧的墙法向量都会保持同一朝向。

如果不做这步，后面 averageNormals() 平均墙法向量时，很容易出现：

  • 有的帧是 n

  • 有的帧是 -n
    最后平均接近 0，彻底失真。

26. 把这帧的地面/墙面参数存起来

C++
// Store (normals are now in camera frame)
ground_planes_.push_back(ground);
wall_planes_.push_back(wall);
cloud_count_++;

含义：

  • 现在这帧已经通过所有检查，并且：

    • ground.normal 已转到 camera frame

    • wall.normal 已转到 camera frame

  • 所以可以把它们加入样本集合

cloud_count_++ 表示：

  • 成功收集到一帧有效点云样本

注意，这里增加计数是在最后才做的，说明前面任何一步失败，这帧都不算有效样本。

27. 发布地面内点云到 RViz

C++
// Publish inlier clouds for RViz debugging (points are in base_link)
{
    sensor_msgs::PointCloud2 out;
    pcl::toROSMsg(*ground_inliers, out);
    out.header.stamp = msg->header.stamp;
    out.header.frame_id = base_frame_;

}

作用：

  • 把拟合出的地面内点发布成 ROS 点云

  • 方便在 RViz 里看“当前地面到底拟合到了哪些点”

逐句看：

27.1 sensor_msgs::PointCloud2 out;

创建输出消息。

27.2 pcl::toROSMsg(*ground_inliers, out);

把 PCL 点云转回 ROS 消息格式。

27.3 out.header.stamp = msg->header.stamp;

继承原始点云时间戳。

27.4 out.header.frame_id = base_frame_;

非常关键：这里内点云是在 base_link 里，因为点云早就变过去了。

27.5 publish(out);

发布出去。

29. 把这帧的关键信息写入调试日志

C++
debug_log_ << "[Cloud #" << cloud_count_ << "] "
           << "ground n_cam=(" << ground.normal.transpose() << ") d=" << ground.d
           << " inliers=" << ground.inlier_count
           << " | wall n_cam=(" << wall.normal.transpose() << ") d=" << wall.d
           << " inliers=" << wall.inlier_count
           << " | dot=" << std::abs(ground.normal.dot(wall.normal))
           << "\n";

含义：

  • 把这帧最终保存的观测结果记录到 debug_log_

记录内容包括：

  • 第几帧有效点云

  • 地面法向量（camera frame）

  • 地面平面 d

  • 地面内点数

  • 墙面法向量（camera frame）

  • 墙面 d

  • 墙面内点数

  • 地面和墙法向量点积绝对值

为什么有用？

因为标定结束后你可以整体打印这些信息，快速判断：

  • 每帧地面法向量是否稳定

  • 墙法向量是否跳变

  • 地面/墙是否接近正交

  • 某些帧内点是否明显偏少

transpose() 只是为了把列向量按一行打印出来，更好看。

30. 打印已收集进度，并检查是否可以开始标定

C++
ROS_INFO("Collected cloud frame %d / %d", cloud_count_, num_frames_);
checkAndCalibrate();

30.1 ROS_INFO(...)

输出当前进度，比如：

  • Collected cloud frame 7 / 30

便于你知道还差多少帧。

30.2 checkAndCalibrate();

调用一个检查函数，通常会判断：

  • 点云帧是否已经够了

  • 激光帧是否也已经够了

如果两个条件都满足，就开始真正的 Ceres 标定流程。

31. 这个函数执行完后，状态到底变成了什么

当 cloudCallback() 成功处理完一帧后，系统里新增了这些信息：

  1. ground_planes_ 里多了一项

    • ground.normal 在 camera frame

    • ground.d 仍是该平面参数

    • ground.inlier_count 表示质量

  2. wall_planes_ 里多了一项

    • wall.normal 在 camera frame

    • 已做跨帧符号统一

  3. cloud_count_ 加 1

  4. RViz 可以看到这帧拟合到的地面/墙面内点

所以这个函数本质不是“直接求外参”，而是在做：

从每一帧点云中提取可用于标定的几何观测量。

32. 用一句话把 cloudCallback 的每个阶段串起来

你可以把它记成这 10 步：

  1. 加锁、防止重复处理

  2. 检查 frame_id

  3. 读取初始 TF

  4. ROS 点云转 PCL

  5. 去 NaN

  6. 点云变到 base_link

  7. 按传感器距离过滤

  8. 体素下采样

  9. 在 base_link 中找地面和墙

  10. 把法向量转回 camera frame 存起来

33. 这段函数里最关键的三个“不要搞反”

最后我把最容易混淆的 3 个点单独拎出来。

第一，点云拟合是在 base_link 中做的

因为要用 z 轴定义“地面”和“墙面”。

第二，存进 ground_planes_ / wall_planes_ 的法向量是在 camera frame

因为优化器要用 R * n_cam 这种形式。

第三，z_min_ / z_max_ 实际不是按 z 坐标过滤

而是按“点到传感器原点的三维距离”过滤。

下一步最自然的是继续把 fitConstrainedPlane() 也逐句拆掉，因为 cloudCallback() 的很多核心其实都埋在那个函数里。
